# 为什么自建一个 Claude Code 中转网关

朋友看到我课程里用 `gw.chengbei.org` 这个域名，问我为什么不直接订 Claude Pro。

"省心啊。一年 240 美元，比你这折腾值多了。"

他说得没错。但我没换。

不是因为我抠。240 美元一年我付得起。是因为这件事根本不是为了省钱——一开始我以为是，做完之后才明白不是。

## 一开始确实是为了省钱

最早的需求很朴素：我自己用 Claude Code、Cursor 这些工具频繁，每月 API + 订阅加起来奔着 50 美元去。

如果只是我一个人用，付了就付了。但还有几个学员、几个朋友。每个人都要付一份 Pro，加起来挺浪费——尤其是大部分人一个月只用几次，但订阅就锁死了你 20 刀。

最朴素的想法：**我能不能用一个共享池子，按用量分摊。**

调研了一圈，发现有现成的开源项目可以做这件事——把 OpenAI / Anthropic 的官方账号包一层，做成多用户、按 token 计费的中转网关。我选了 sub2api（CRS 2.0 那个）。

一台日本 VPS，月费 ~$10。装上 sub2api，用 Caddy 反代到 `gw.chengbei.org`。

完工那天我算了一下：我一个人 + 4 个朋友共用，月成本人均 ~$2 美元。

省钱目标达成。但这不是这件事最让我满意的点。

## 真正让我离不开的，是控制权

第一周用下来，我开始体会到一些**商业方案永远给不了**的东西。

**第一件**：我能看见每个人用了多少 token。Pro 订阅是个黑盒，你只知道"今天 limit 还没到"。自建的话我能看到——这位朋友昨天跑了 200 万 token 在调一个奇怪的 prompt 循环，那个学员每天稳定用 5 万。

看清楚之后，我能做更细的事：给学员配 token 上限，给朋友按月预付，给我自己留无限。商业方案做不到——它只给你一个统一的"Pro"。

**第二件**：我能控制接哪个供应商。我不是只跑 Anthropic 的——加上 OpenAI、Google、DeepSeek、Kimi，谁有便宜或者强项就用谁。模型切换在 API 层而不是产品层完成，下游应用感觉不到。

**第三件**：稳定性。Claude Pro 在大陆访问偶尔抽风（IP 段被风控、Cloudflare 拦截、登录态过期），订阅用户只能干等。自建网关跑在我自己的 VPS 上、走我自己的 IP、用我自己的证书。坏了，我自己修。

不是说商业方案不好。是商业方案的边界**永远是产品经理画的**，不是你画的。

## 关于"自建"这件事

很多人对自建有误解——以为是"为了省钱"或者"为了酷"。

其实自建在我这里就一件事：**我对这个工具的控制权，能延伸到哪一层。**

商业产品给你一个表层。你能改的只是表层的几个按钮。背后的逻辑——多少配额、谁能用、用什么模型、出错时降级到哪——都是别人决定的。

自建意味着这些决定权回到你手里。你成了**产品的设计者**而不只是使用者。

这件事的代价：你得维护它。要应对官方 API 变化，要处理凭证轮换，要看监控、改配置、查日志。我每个月大概花 1-2 小时在这上面，不算多但也不是零。

收益：当我需要做一个新工具、新课程、新实验，**那个底层的中转网关就在那儿**，随时给我接。这种"我有自己的基础设施"的安全感，是 Pro 订阅给不了的。

## 给独立开发者的一句话

独立开发者最稀缺的东西不是写代码的能力——AI 时代写代码已经很便宜。

最稀缺的是**"我可以自己搞一套"** 的判断力。看见一个商业产品，能立刻判断它的边界在哪、自己接管要付出什么、值不值得接管。

这个判断力不是靠看文章训练出来的。是靠**真的接管过几次**练出来的。

`gw.chengbei.org` 这个域名背后那台 VPS，对我来说远不止是省了几百块订阅费。它是我"我可以自己来"的一个具体证据——证据放在 DNS 里、放在 Caddy 配置里、放在我每月的几条 cron job 里。

下次有个商业方案让你不爽，先别急着付钱。

问问自己：**这套东西的核心其实就那么几个步骤，我自己能不能搭一套？**

很多时候答案是"能"。能而不为，是另一种选择。但只有亲手能为过的人，"不为"才是有意义的。
